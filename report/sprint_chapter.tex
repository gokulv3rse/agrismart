\chapter{Sprint Planning and Execution Methodology}

This chapter documents the Agile sprint methodology adopted for the development of AgriSmart. The project was structured into three two-week sprints, each with clearly defined goals, user stories, functional deliverables, architectural decisions, UI designs, test cases, daily progress logs, velocity tracking, and retrospective reviews. The sprint-based approach allowed the team to iteratively build and validate features, ensuring continuous alignment with the project's core objectives.

The overall system, shown in Figure~\ref{fig:system_arch}, follows a six-block data flow from user input through image upload, AI inference, decision engine processing, platform action management, and final output delivery.

\subsection{Functional Document}

The primary functional deliverable of Sprint 1 was the construction and training of two deep learning models — one for insect/pest classification and one for rice disease classification — along with the Flask-based inference server that serves predictions in response to image uploads from the React frontend.

The \textbf{insect classifier} is based on the ResNet18 architecture, loaded with ImageNet-pretrained weights (\texttt{IMAGENET1K\_V1}) to leverage rich visual feature representations. The model's fully connected head was replaced with a custom Sequential block comprising a Dropout layer (rate: 0.4) followed by a Linear layer mapping from ResNet18's 512-dimensional feature space to five target pest classes. All backbone parameters were frozen and only the final residual block (\texttt{layer4}) was unfrozen for gradient updates during the initial phase. The Adam optimizer was used with a learning rate of 0.001. Training proceeded for 30 epochs, with model checkpoints saved whenever the validation accuracy improved. The best validation accuracy recorded was \textbf{87.24\%}.

The \textbf{rice disease classification model} employs the EfficientNet-B0 architecture, also initialized with ImageNet weights. All layers were unfrozen during fine-tuning — an approach appropriate for small-to-medium datasets where full fine-tuning outperforms frozen-backbone approaches. The classifier head was replaced with a Sequential block containing Dropout (rate: 0.5) and a Linear layer projecting to six target rice disease classes. Best validation accuracy achieved was \textbf{86.06\%}.

Both models are served through a unified Flask inference server. Upon receiving an image, the server preprocesses it through a standardized pipeline: resizing to 224$\times$224 pixels, applying ImageNet normalization (mean: [0.485, 0.456, 0.406], std: [0.229, 0.224, 0.225]), converting to a PyTorch tensor, and executing a forward pass within a \texttt{torch.no\_grad()} context. Softmax probabilities are computed, ranked, and returned as JSON to the React frontend.

\subsection{Architecture Document}

Sprint 1 established a clean three-tier architecture:

\begin{itemize}
    \item \textbf{Client Tier:} React application with image upload (drag \& drop), model selector toggle (Pest / Disease), and Analysis Report output panel.
    \item \textbf{Server Tier:} Flask inference server (:5001) with two model checkpoints loaded in memory. POST requests routed by model type; responses return ranked class probabilities.
    \item \textbf{Data Tier:} Supabase PostgreSQL (\texttt{diagnoses} table) and Supabase Storage (\texttt{diagnosis-images} bucket). Row Level Security (RLS) ensures user-scoped access.
\end{itemize}

\subsection{Sprint Retrospective}

\textbf{What went well:}
\begin{itemize}
    \item Both CNN models (ResNet18 and EfficientNet-B0) were successfully trained and integrated into the Flask inference server within the sprint window, with best validation accuracies of 87.24\% and 86.06\% respectively.
    \item The React-to-Flask image upload pipeline worked correctly on the first integration test, with no CORS or encoding issues.
    \item Supabase authentication and RLS policies were configured correctly, ensuring user-scoped data access from day one.
\end{itemize}

\textbf{Improvement areas:}
\begin{itemize}
    \item Initial model serving loaded checkpoints on every request, causing latency. Mitigation: checkpoints are now loaded once at server start and kept in memory.
    \item The decision rules mapping (class label $\rightarrow$ chemical recommendation) was hardcoded in the frontend JavaScript. This was identified as a technical debt item to be addressed in Sprint 2 by migrating rules to a Supabase database table.
\end{itemize}

\subsection{Functional Document}

Sprint 2 transitioned the platform from a single-page prototype to a multi-page full-stack application with persistent state, user authentication, and data-driven features.

\textbf{Express API Server:} A Node.js 20 / TypeScript Express server was introduced as the backend API layer (port 3000). All routes are protected by a custom \texttt{requireAuth()} JWT middleware that validates the Supabase-issued access token on every request. Routes implemented include \texttt{/api/plants} (CRUD), \texttt{/api/schedules} (create, list, mark-complete), and \texttt{/api/weather} (OpenWeatherMap proxy). The Vite development server proxies all \texttt{/api/*} requests to the Express server, eliminating CORS issues during development.

\textbf{Analytics Dashboard:} The \texttt{Analytics.tsx} page queries the \texttt{diagnoses} table and renders: a Disease Frequency bar chart (Recharts), a monthly trend line chart, an action distribution summary, and aggregate statistics (total diagnoses, action required count, healthy count).

\textbf{Spray Schedule Manager:} The \texttt{SpraySchedule.tsx} page consumes the \texttt{/api/schedules} endpoint. Each schedule stores the product name, dosage, interval in days, total applications, completed applications, next spray date, and status (\texttt{active} / \texttt{completed}). Clicking ``Mark Done'' triggers \texttt{PATCH /api/schedules/:id/complete}, incrementing the counter and recalculating the next spray date.

\textbf{Weather Advisory:} The \texttt{WeatherWidget} component calls \texttt{/api/weather} and displays current temperature, humidity, and wind speed. An advisory text reading ``Not ideal for spraying'' is shown when rain probability exceeds 40\% or wind speed exceeds 15 km/h.

\subsection{Architecture Document}

Sprint 2 added the Express API server tier between the React frontend and Supabase, enabling server-side JWT validation, weather proxying, and structured CRUD operations. The data flow became:

\begin{enumerate}
    \item React client authenticates via Supabase Auth and stores the JWT in the Zustand auth store.
    \item All API requests include the JWT in the \texttt{Authorization: Bearer} header.
    \item Express \texttt{requireAuth()} middleware validates the token against Supabase before executing route handlers.
    \item Route handlers use the Supabase JS SDK (server-side with service role key) to query PostgreSQL.
\end{enumerate}

\subsection{Functional Document}

\textbf{Multi-Model Inference Architecture:} Sprint 3 consolidated three independently hosted AI model pipelines into a unified Express inference router (\texttt{/api/roboflow/infer}). The router selects the model endpoint based on the \texttt{modelId} parameter:

\begin{itemize}
    \item \texttt{agrismart/1} $\rightarrow$ Local MobileNetV2 Flask server (:5001) — Tomato and Potato diseases (7 classes)
    \item \texttt{hf-rice/1} $\rightarrow$ HuggingFace Space: \texttt{sanchaikb-fertilizer-model.hf.space/predict} — 6 Rice disease classes
    \item \texttt{hf-insect/1} $\rightarrow$ HuggingFace Space: \texttt{SanchaiKB-Insect-Classification-Model.hf.space/predict} — 5 Pest classes (EfficientNet-B0)
\end{itemize}

The response from each model is normalized by the \texttt{normalizePredictions()} function in \texttt{decisionEngine.ts}, which handles both array-based (local model) and dictionary-based \texttt{all\_scores} (HuggingFace EfficientNet) response formats. Temperature Scaling ($T = 1.8$) is applied in the local server to calibrate confidence scores and prevent overconfident predictions.

\textbf{Decision Engine:} The \texttt{buildDecision()} function implements a deterministic rules engine that:
\begin{enumerate}
    \item Checks if the top label matches a known healthy class pattern.
    \item Queries the \texttt{spray\_recipes} table for a recipe matching \texttt{model\_id} and \texttt{class\_label}.
    \item Validates that confidence $\geq$ \texttt{min\_confidence} threshold.
    \item Returns a structured decision object: \texttt{\{spray, actionType, label, confidence, recommendation, dosage, notes\}}.
\end{enumerate}

\textbf{Spray Recipes Database:} A master migration (\texttt{0008\_master\_recipes.sql}) inserts 18 spray recipes across all three models: 7 for \texttt{agrismart/1}, 6 for \texttt{hf-rice/1}, and 5 for \texttt{hf-insect/1}. Each recipe specifies the chemical product, dosage (e.g., 1.5ml Chlorpyrifos per litre of water), action type (pesticide/fungicide), minimum confidence threshold, and special field notes.

\textbf{Admin Decision Rules Panel:} The \texttt{Rules.tsx} page provides a CRUD interface for spray recipe configuration. Administrators can enable/disable rules per class, update the recommended chemical and dosage, and adjust the confidence threshold — without requiring any SQL knowledge or backend access.

\textbf{PDF Report Generation:} The \texttt{reportGenerator.ts} module uses jsPDF and jsPDF-AutoTable to generate a downloadable PDF containing: the diagnosis date, crop image thumbnail, detected condition name, confidence score, AI recommendation, dosage, treatment type, and special notes.

\subsection{Sprint Retrospective}

\textbf{What went well:}
\begin{itemize}
    \item The unified multi-model routing system worked correctly on first integration — all three models (local MobileNetV2, HuggingFace Rice, HuggingFace EfficientNet-B0) returned results correctly from the single \texttt{POST /api/roboflow/infer} endpoint.
    \item Temperature Scaling calibration visibly improved confidence score realism — predictions that previously returned 100\% confidence now return calibrated scores in the 70--95\% range, reducing overconfidence artifacts.
    \item The decision engine's 3-state badge (Action Required / Healthy / Review) correctly distinguished between genuinely healthy crops, disease detections with available recipes, and detection with missing recipe configuration.
    \item The Admin Rules Panel allowed non-technical users to modify spray recommendations without any DB access — a key usability milestone for the agricultural use case.
\end{itemize}

\textbf{Improvement areas:}
\begin{itemize}
    \item HuggingFace free-tier Spaces experience a 30--60 second cold start after periods of inactivity. This is a platform limitation requiring a paid tier or a keep-alive ping mechanism as future work.
    \item The IoT sprinkler module is currently simulation-only. Physical integration with a Raspberry Pi via MQTT broker is identified as the highest-priority future enhancement.
    \item The local MobileNetV2 model was trained on controlled lab-condition PlantVillage images. Real-world generalization for complex field backgrounds remains an open research challenge to be addressed via data augmentation and domain adaptation in a future sprint.
\end{itemize}
